# extraction-playbook — 从既有材料推测功能的启发式手册 / Static-Recon Extraction Playbook

> engineer-legacy-recon **阶段 1–3** 的核心方法。SKILL.md 定义了流水线与三层可信度（明示/推断/缺口），本文件给出把"一堆页面文字和菜单"读成"实体·字段·操作·状态机·角色"的具体启发式。
>
> 一句话原则：**页面上每一个可见元素都是一条功能线索**。表头是字段，按钮是操作，标签页是状态，筛选栏是查询，统计头是报表，"共 N 条"是列表规模。你的工作是把这些线索翻译成结构化模型，并诚实标注哪些是写着的、哪些是猜的、哪些没提。

## 1. 导航树解析（阶段 1）

导航菜单是**白给的路由地图**。把缩进/层级直接翻译成账本骨架：

- **顶层菜单** → 一级模块（如 `cms管理`、`机构管理`、`考试管理`、`合格证管理`、`财务`、`统计`）。
- **二级菜单** → 子模块或功能域（如 `考试管理 → 飞行证考试`）。
- **叶子菜单** → 一个视图页面（如 `理论管理`、`成绩管理`、`考次链接`）——每个叶子建一个账本 node，`extraction_status = "seeded"`。
- **重复出现的叶子子结构**（如飞行证/工程师证/航拍证/青少年证下都有"理论管理/实践管理/成绩管理/考次链接"）→ 识别为**参数化的同构模块**：一套页面模板 × 证件类型枚举。这能极大压缩数据模型（一张考试表 + type 字段，而非四张表）。

**产出**：`模块 → 子模块 → 页面` 树，写入账本。此步几乎全是 `明示`（菜单名白纸黑字），层级归并关系可能是 `推断`。

## 2. 页面级要素提取（阶段 2）

对每个页面材料，逐类抽取。下面每一类都给出「线索 → 推断什么 → 典型可信度」。

### 2.1 列表表头 → 实体字段

一个列表页的表头列，几乎就是该实体的字段清单。

- `序号 / 姓名 / 身份证号 / 手机号 / 培训机构 / 微信绑定` → `学员` 实体的字段。
- 列名即字段名，列的取值样例暗示类型：`身份证号` 18 位串→字符串；`添加时间 2026-03-26 15:31:12`→datetime；`微信绑定=未绑定`→枚举/布尔。
- **可信度**：字段**存在** = `明示`；字段**类型/必填性** = `推断`（材料只给了展示值，没给 schema）。

### 2.2 筛选栏/搜索框 → 查询参数 + 枚举取值

页面顶部的下拉、输入框、日期范围就是列表的查询条件，下拉的**选项本身**往往泄露枚举全集。

- `请选择属性 / 请选择机型 / 请选择省份 / 请选择城市 / 只看考点 / 请输入机构名称 / 请选择状态` → 机构列表的过滤维度。
- 标签页 `初审/已通过/已驳回/变更审核/已过期/全部` 出现在筛选区 → `审核状态` 字段的**枚举全集**（`明示`，因为选项被列出来了）。
- **可信度**：过滤维度存在 = `明示`；未展开的下拉（如"请选择省份"没列出所有省）= 取值集为 `推断/缺口`。

### 2.3 操作列/按钮 → 操作动词 + 状态转移

行内操作列和页面按钮是**操作（用例）**的直接证据，且常常暗示状态机。

- `编辑 / 恢复` 与 `编辑 / 禁用` 成对出现 → 机构有 `启用⇄禁用` 状态，`恢复`是启用的逆，即**软删除/停用状态机**（`推断`，从按钮成对性反推）。
- `详情 / 一键打印 / 作废` → 证件打印流程有 `作废` 转移。
- `查看 / 调整 / 记录` → 考次可调整，且有操作审计（`记录`）。
- `新建 / 修改 / 查看 / 批量导入 / 批量导出` → 标准 CRUD + 批量能力。
- **可信度**：按钮**存在** = `明示`；按钮触发的**状态转移语义** = `推断`。

### 2.4 标签页/分段控件 → 状态机的状态集

页面内的分段（Tab）通常是同一实体按状态分桶。

- 理论管理页 `待审核 / 未通过 / 已通过` → 考试审核对象的状态机三态。
- 考试申请页 `新建申请 / 已申请 / 待考试 / 未通过` → 考试申请状态机。
- 发货管理 `全部/待发货/已下单/已揽件/运输中/派送中/已签收` → 物流状态机（7 态，`明示`——标签被逐个列出）。
- **可信度**：状态集 = `明示`；状态间**合法转移** = `推断`（画成状态机图时标注箭头为推断）。

### 2.5 统计/汇总头 → 聚合指标与报表

列表上方的数字块是聚合报表需求。

- 合格证列表头 `合格证总数：227202 / 首次取证：7351 / 增发：121415 / 更新：17 / 免考：53601 / 补发：1 / 即将到期：11266` → 按 `订单类型` 维度的分组计数 + `即将到期` 派生指标。
- 发票管理 `总开票金额 / 未开具金额 / 增发/金额 / 更新/金额 …` → 财务聚合看板。
- **可信度**：指标**存在** = `明示`；计算口径（如"即将到期=近一个月内"，材料有时会写明）= `明示或推断`。数据量级（227202）= `明示`，是数据规模与分页/性能设计的重要输入。

### 2.6 表单字段 → 输入模型与校验

新建/编辑页、详情页的字段是写模型。

- 创建考场：`民用无人机驾驶员考试点 / 所属区域 / 考场类型(固定|临时) / 可报名考试种类(多选)` → 考场实体写模型，`考场类型`为枚举，`可报名考试种类`为多选关联。
- 发票抬头：`发票种类(全电专票|全电普票) / 单位名称 / 纳税人识别码 / 详细地址 / 注册电话 / 开户银行 / 银行账户 / 邮箱` → 完整开票主体模型。
- 注册页给出的规则文案（"密码长度6-20位，需包含大小写字母、数字和特殊符号"）→ **校验规则（`明示`，因为文案写出来了）**。未写出文案的字段校验 = `推断/缺口`。

### 2.7 分页/计数 → 列表规模与分页存在性

`共 75895 条`、`共 0 条记录，第 1/1 页，每页 20 条`、`12345>»` → 列表分页存在、每页 20、数据量级已知。影响后端分页 API 与索引设计（`推断`）。

### 2.8 面包屑/侧栏高亮 → 页面归属与跳转

材料里重复出现的侧栏（cms管理/机构管理/…）确认了每个页面的模块归属；`列表→详情→编辑`的按钮链确认页面跳转图。

## 3. 实体归并与状态机推断（阶段 3）

### 3.1 跨页归并同一实体

同一实体在多页以不同切面出现，必须归并为一个数据模型，取字段并集：

- `学员` 出现在：学员列表（姓名/证件号/手机/机构/微信绑定）、成绩管理（理论/实践成绩）、缴费、证件详情（合格证信息/签发日期/起止日期）、考生状态（账号/状态/锁定原因）。→ 归并出 `学员` 主实体 + `成绩`/`证件`/`缴费`/`考生账号` 关联实体。
- 归并线索：**同为身份证号主键**的行属于同一人。
- **可信度**：归并关系 = `推断`（材料没画 ER 图，是你据主键与语义反推的）。

### 3.2 反推关系基数

- 一个 `机构` 有多个 `学员`（学员列表有"培训机构"列）→ 机构 1:N 学员。
- 一个 `机构` 可有多种 `培训资质`（机构信息里一行挂多个"AOPA…训练机构"标签）→ 机构 N:M 资质类型。
- 一个 `考场` 支持多种 `考试种类`（可报名考试种类多选）→ 考场 N:M 考试类型。
- **可信度**：全部 `推断`，在 CONTEXT.md 数据模型里注明。

### 3.3 画状态机

把 2.3/2.4 的证据连成状态机。示例（考试申请，箭头为推断）：

```
草稿 → 已提交 → 审核中 → 审核通过 → 已安排考试 → 考试完成 → 成绩已同步
                  └→ 审核驳回 → 草稿（可重新提交）
```

状态集来自标签页（`明示`），转移箭头来自按钮语义与业务常识（`推断`）。

## 4. 角色与多端划分

遗留系统常有多端，材料的顶部标识和菜单差异会泄露角色：

- 材料标 `无人机合格证管理系统(管理端) superadmin` vs `(机构端) 手机测试` → 两个端 + 两类角色。
- 管理端菜单（机构审核/考试管理/合格证管理/财务/统计/日志）= AOPA 协会管理员视角。
- 机构端菜单（机构资质/学员管理/考试申请/缴费/权限管理）= 培训机构视角。
- 机构端还有 `权限管理 → 角色管理/管理员管理` → 机构内部还有二级 RBAC。
- **产出**：角色 → 可见模块矩阵，写入 REQUIREMENTS.md 的角色旅程。**可信度**：端/角色存在 = `明示`；完整权限矩阵 = `推断/缺口`（材料只示范了部分账号）。

## 5. 可信度分级的判定规则

拿不准时**往下沉**（宁可标弱不可高估）：

| 你手里的证据 | 标 |
|:------------|:--|
| 菜单名/表头/按钮/标签/统计数字/写出来的校验文案，白纸黑字 | `明示` |
| 从明示证据 + 业务常识合理反推：字段类型、关系基数、状态转移、软删除语义、未展开枚举 | `推断` |
| 材料完全没提：后端算法、精确权限矩阵、支付内部、定时任务、字段级校验（无文案）、外部集成内部 | `缺口` |

- `明示` 与 `推断` 的分界：**信息本身**被写出来了是明示；**信息之间的关系/语义**是你补的就是推断。
- 任何 `缺口` 都要进 `RECON-FIDELITY.md` 的缺口清单并（normal 模式下）向用户追问。

## 6. AOPA 实战示例（浓缩）

用户贴出 AOPA 无人机合格证管理系统的管理端+机构端全部页面。侦察产出片段：

- **导航树**（阶段 1，`明示`）：管理端 8 大模块，其中"考试管理"下 5 种证件（飞行证/工程师证/航拍/青少年/应用技能）各含"理论/实践/成绩/考次链接"→ 识别为**参数化同构模块**（考试模板 × 证件类型枚举）。
- **实体**（阶段 3，`推断`归并）：`机构`、`学员`、`考试申请`、`成绩`、`合格证/证件`、`考场`、`订单`、`发票`、`发货单`、`试题/题库/试卷`、`日志`。
- **状态机**（`明示`态 + `推断`转移）：机构审核（初审/已通过/已驳回/变更审核/已过期）、证件打印（初始/待打印/打印中/待校验/作废）、物流（7 态）。
- **报表**（`明示`）：证件统计、考试统计（通过率 92.81%）、训练机构统计、财务应结汇总、机构核销查询。
- **缺口**（`缺口` → 追问用户）：支付渠道内部、发票开具对接的税务系统、快递单号对接的物流 API、理论考试系统的判分算法、各字段精确校验规则。

这份侦察结果即可作为 `REQUIREMENTS.md` 的骨架，交 engineer-requirements/architect/job 继续。
