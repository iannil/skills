# Causal Chain Analysis — 因果链分析（RC 版）

> **来源声明**: 因果链方法源自现代 TRIZ 的因果链分析（Cause-Effect Chain Analysis, CECA）；其被本 skill 用作**方法内核**的 RC（观察收敛）哲学框架来源于 [zhurongshuo.com](https://zhurongshuo.com)。

因果链分析是深入挖掘问题根本原因的系统性工具：从用户的**表层问题**出发，反复追问"为什么"，层层深入，建立一条（或一张网状的）因果链，以便识别**关键缺点**、把它转化为可解决的**关键问题**。

本 skill 与普通因果链分析的根本区别在于：它把整个方法**架在 RC 理论之上**——不是收尾时点缀几句哲学，而是让 RC 公理**决定每一步为什么这样做**。先读懂下面这一节，后面的每个动作才有根。

---

## 一、RC 根基：一条因果链，就是一次观察收敛

这不是类比，而是 RC 存在论的直接主张——**因果本身是"收敛"的一种方式**（A1 基底优先）。当你为某个缺点画出一条因果链，你是在把一个高维的**可能性基底**，沿某个方向**收敛**、**锁定**成一棵低维的确定性的树。

这一句话，直接推导出本方法的全部纪律：

| 方法里的动作 | RC 内核（它的**理由**，不是脚注） |
|---|---|
| 整条链是**地图，不是疆域**；换个观察者或口径，会收敛出**不同的**链 | A1 基底优先 + 认知论**理论降维**——链是对基底的一次降维投影，必然有损、有立场 |
| 每加一个节点，必须**即时校验**才算数；没校验的标 `[假设]` | A3 **观察锁定**——确定性只在"被观察"那一刻锁定；未经校验的原因，仍是**可能性伪装成事实** |
| 拒绝"一次性写完整棵树再回头修剪" | A6 **共识强化**——一次锁定整棵树 = **过早共识**，把一堆猜测互相印证成"客观现实"；逐节点展开是对它的刻意抵抗 |
| 判定一个节点是**末端缺点**、停止展开 | A7 **余量守恒**——真末端 = 此处**已无可用余量**（自然律/物理约束）；要和"余量在别人手里"的**边界末端**分开 |
| 识别**关键缺点** | A7——关键缺点 = **可用余量最大 × 杠杆最高**的节点；解决它，就是在那里**重新收敛**出另一种未来 |
| 组织类问题里反复出现的高杠杆节点 | 秩序论 **权力 / 评估异化**——"决策者不承担后果"是可预测的关键缺点原型 |
| 分析者自己也可能是链上一个节点（如"创始人的焦虑"） | A4 **观察者—被观察者对称**——观察者从不在系统之外 |
| 多个关键问题里**先打哪一个** | 实践论 **可持续决策**——优选那个保留最多可用余量、避免不可逆失败的 |

**贯穿全程的三个 RC 追问**（每展开一步都在心里问）：
1. **（A3）** 这个节点是我**观察锁定**的，还是还停在**可能性**？——没锁定就标 `[假设]`，别假装确定。
2. **（A6）** 我是不是**过早收敛**了？——还有哪些并列的、未被展开的**余量分支**被我忽略？
3. **（A7）** 这个"到底了"是**真的没余量**，还是只是**超出了我的观察边界**（余量握在别的权力节点手里）？

## 二、工作流程 / Workflow

### 阶段 0：锚定初始缺点 = 选定收敛方向 / Choose the convergence direction

用户给你的是**表层缺点**（surface disadvantage）——他直接感受到、不满意的现象。你的第一步不是急着追问"为什么"，而是先决定这次观察要往哪个方向收敛——即确定**初始缺点**（initial disadvantage）。

> **RC 依据（A1）**：同一堆事实，选不同的初始缺点，会收敛出**完全不同的树**。所以"选口径"不是走过场，而是决定整次降维投影的方向。必须先锚定、并与用户确认，否则后面挖得越深、离错的目标越远。

- **表层缺点**：用户原话，常笼统、掺情绪、或已隐含某个解法。例："我们 App 用户留存太差。"
- **初始缺点**：本质上要被降低的那个负面目标状态，用规范表述写出。例："新用户在首周流失率过高。"

做三件事（通常各一两句），然后**与用户确认**再往下：

1. **提炼**：把表层问题重述为一条规范的缺点陈述（见《缺点表述规范》）。若有多个合理口径，给 2 个候选让用户选——这就是在明确"往哪个方向收敛"。
2. **划定边界（用可用余量判定）**：哪些因素**你手里有可用余量**（可改变、在项目内，继续深挖）？哪些是**你无余量**的——自然律、或余量握在你边界之外的权力节点手里（将来成为末端）。
3. **确认口径**：跟用户核对"我们要把哪一个负面状态降到最低"。

### 阶段 1：逐节点展开 = 逐步观察锁定 / Grow by locking one node at a time

初始化队列 = { 初始缺点 }。然后循环：

**① 取节点**：从队列取出优先级最高的节点。优先级 = 该节点当前**对初始缺点的影响程度**（越靠主干、越可能藏着关键缺点的，越优先）。
> **RC 依据（A6）**：优先展开主干、而非把每条枝节均匀追到底，是为了**把观察力集中在最可能重新收敛的地方**，同时避免一次性铺开全树造成的过早共识。

**② 追问直接原因**：问"**是什么直接导致了它？**"——只往下**一层**，列出直接原因，不跳级（跳级会漏掉中间可干预的、有余量的节点）。

**③ 标注逻辑关系**：多个原因之间是——
   - **AND（与）**：几个原因**同时**存在才导致该缺点（缺一不可）。
   - **OR（或）**：**任一**原因单独即可导致该缺点。
   分清 AND/OR 极重要：它决定了将来"切断"这个缺点、需要在几处**重新收敛**。

**④ 就地观察锁定每个新节点**（本方法的灵魂 = A3 的直接体现，见《三项即时校验》）。任一项不过关，**立即修正**；若发现是上层节点表述有误，**回退**去改上层。
> **RC 依据（A3）**：一个没被校验的原因，不是"确定的因果"，只是**尚未锁定的可能性**。逐节点校验，就是逐个把可能性**观察锁定**成确定——把错误消灭在它产生的那一层，因为错误会沿链**向下放大**（认知论：降维一旦在上层失真，下游整片作废）。

**⑤ 判定末端或入队**：
   - 若该原因是**末端缺点**——**注意区分两种末端**：
     - **真末端（无可用余量）**：自然规律、物理约束、数学必然。标 `⊗自然律`。到此为止，不必强攻（A7：此处基底已无未锁定的余量）。
     - **边界末端（余量在别处）**：本身可变，但改变它的**收敛能力握在你项目/权力边界之外**（外部环境、上级决策、他方）。标 `⊗边界`。它虽停展开，却常是提示"真正的权力节点在哪"的重要信号。
   - 否则，把它**按优先级放回队列**。

**⑥ 回到 ①**，直到队列为空。

> **组织类因果链的专项提示（秩序论）**：当链落在团队/组织/协作上，主动去找**评估异化**节点——"做决策的人不承担后果"。RC 预测这类节点是恶性循环的源头，因此往往正是**高杠杆的关键缺点**。同时留意 A4：分析者本人（如创始人的状态、你自己的决策）常常也是链上一个真实节点，别把自己排除在系统之外。

### 阶段 2：识别关键缺点 = 定位可用余量最大处 / Locate maximal available margin

链建成后（或用户随时要求时），做一次**可用余量扫描**：

- **关键缺点（key disadvantage）** = 同时满足——(a) **高杠杆**：切断它能同时消除多条下游分支（常是多条分支**共享的汇聚节点**）；(b) **可用余量大**：它落在你**仍能重新收敛**的区域，不是末端的自然律。
  > **RC 依据（A7）**：余量守恒保证——无论一个缺点被多少既成事实"锁死"，总有未锁定的余量存在；关键缺点就是那个**余量最丰、又最省力**的再收敛点。
- 把关键缺点**反转**为**关键问题**：例"重复计算导致响应慢"→"如何在不重复计算的前提下得到结果？"
- 通常给 1–3 个候选关键缺点 + 理由（杠杆 × 可用余量），交用户选。

**若给出多个关键问题、需建议先打哪个**，用实践论的**可持续决策**准则排序：
> 优先那个**保留最多可用余量、且能避免不可逆失败**的关键问题——即使它见效不是最快。因为最优决策的核心是"保持选择权在场、让游戏留在'还能再来一轮'的状态"（实践论），而不是一把梭哈式地锁死系统。

## 三、缺点表述规范 / How to state a disadvantage

一个"缺点"是某功能或参数处于**不足、过度、有害、缺失**的负面状态。表述必须让下一步追问"为什么"能自然进行。

| 要求 | 说明 |
|------|------|
| **是负面状态，不是中性事实** | ✅"电池续航过短" ❌"设备使用锂电池"（中性，无法追问为什么不好） |
| **不要把解决方案伪装成缺点** | ❌"缺少缓存机制"（这是解法的缺失）✅"相同请求被重复计算，响应慢"（缓存是将来的解，不是原因） |
| **一个节点只说一个缺点** | 把"又慢又容易崩"拆成两个节点，各自追因 |
| **可观察、尽量可度量** | "首屏加载 > 5 秒"优于"感觉很卡" |
| **对准参数而非笼统抱怨** | "客服响应时长过长"优于"客服体验差" |

> **RC 提示（A3）**：表述规范之所以重要，是因为只有一个**清晰、可观察**的负面状态，才谈得上"被观察锁定"。一句含糊或掺解法的表述，根本无法被验证——它永远停在可能性，链就断了根。

## 四、三项即时校验 / The three per-node checks

每加入一个节点，立刻过三关——**这三关就是 A3「观察锁定」的具体操作**：把一个候选原因，从"可能性"锁定为"确定节点"。任一关不过 → 就地修正，甚至回退改上层。

### 1. 表述是否规范？ / Well-formed?
按《缺点表述规范》检查。最常见错误：把"解决方案的缺失"写成原因，或写成中性事实。不规范 → 就地重写。

### 2. 拆分是否**直接且完备**？ / Direct AND complete?
- **直接**：列的是**紧邻的下一层**原因，中间没跳过环节。自检："从这原因到该缺点，还能不能再插一个'为什么'？"能，就是跳级了，补回中间节点。
- **完备**：这些原因**合起来**是否足以解释该缺点？有没有遗漏的并列原因？
  > **RC 提示（A6）**："完备"这一关，正是在对抗**过早共识**——先把并列的余量分支都摊出来，再判断哪些是末端，而不是抓住第一个看着顺眼的原因就锁定收敛。宁可先列全，再收敛。

### 3. 是否与已有节点重复？ / Duplicate?
扫一遍链上已有节点：
- 若这个原因**已在别处出现**，不要新建重复节点——让多条分支**指向同一个共享节点**（因果链因此常是网状而非纯树）。
  > **RC 提示（A7）**：这种被多条分支共同指向的**汇聚节点**，往往正是"可用余量 × 杠杆"最高处——是关键缺点的头号候选。
- 若表述近似但实质不同，澄清差异，拆成两个精确节点。

## 五、输出格式 / Output format

用**缩进树**呈现，边生长边展示进展。每个节点标注：编号、逻辑关系、末端类型。

```
[0] 初始缺点：新用户首周流失率过高
 └─(AND)─ [1] 新用户首次使用时未获得核心价值
 │         ├─(OR)─ [2] 关键功能藏得太深，用户找不到
 │         │         └─ [4] 首次引导流程缺失 …（有可用余量，待展开）
 │         └─(OR)─ [3] 首次加载 > 8 秒，用户未进入就离开
 │                   └─ [5] 首屏一次性拉取全部数据  ★关键缺点候选（余量大×高杠杆）
 └─(AND)─ [6] 竞品同期做免费推广            ⊗边界（收敛能力在外部，余量不在我手）
```

- `★` = 关键缺点候选（注明"余量 × 杠杆"）；`⊗自然律` / `⊗边界` = 两类末端并注明原因；`[假设]` = 尚未观察锁定、待验证。
- 展开过程中，简要旁白你在展开哪个节点、某次校验触发的修正、以及某个 RC 追问（A3/A6/A7）如何影响了判断——让用户看到链是"边收敛边锁定"的。

收尾给出：
1. 完整因果链（树）。
2. 1–3 个**关键缺点候选** + 理由（可用余量 × 杠杆）。
3. 反转出的**关键问题**；若多个，按可持续决策准则给出建议顺序。

## 六、交互原则 / Interaction principles

- **信息不足就问（A3）**。追因需要事实；某层你只能靠猜时，停下来问用户"这一步的直接原因，据你所知是哪些？"——而不是编造分支假装已锁定。
- **让用户参与判断优先级与末端边界（A7/A4）**。"哪些是我无余量的外部约束"往往取决于用户的项目边界与权力位置，应与用户确认，而非替他拍板。
- **可增量**。用户可能只想先展开主干几层；随时可返回队列继续。
- **双语**：用用户的语言回应；关键术语给中英对照，如**"可用余量 (available margin)"**。
- **诚实与自省（A4/认知论）**：证据薄弱的分支明确标 `[假设，需验证]`，不伪装成已确证的根因。也提醒自己：这条链只是**一次**降维投影，不是唯一真相——必要时提出"换个初始缺点口径，会收敛出另一条链"。
