# 产品分析框架

## 基本原则

这个框架来自对 20+ 个真实赚钱产品的拆解。核心信念：

**产品不是在卖功能，是在卖"少做一件烦人的事"。**

分析时不要停留在表面描述。每次都要追问：这个产品真正在帮用户减少什么痛苦？用户不是为功能付钱，是为"少亏钱、少浪费时间、少错过客户"付钱。

**别做 AI 玩具。做老板愿意报销的止痛药。**

---

## 分析流程

遵循以下 8 个维度，按顺序输出。每个维度都需要给出有洞察力的判断，而不是泛泛的描述。

### 1. 一句话本质

用最精炼的一句话概括这个产品的本质。句式参考：

> 它的本质是：[核心定位]

关键：不要描述功能，要描述**它对用户意味着什么**。

**好例子：**
- NotFair：把 AI 广告优化变成"帮你少浪费广告预算的代理"
- Marky：把作者讨厌但必须做的社交营销变成自动化流水线
- Mommy：给新手父母提供控制感（不是提供数据记录）
- Gear Snap：用 AI 识别高错误成本物品（不是汽车零件扫描）

**差例子：**
- NotFair 是 AI Google Ads 管理工具（这只是功能，不是本质）

### 2. 用户是谁

列出核心用户群体，每个群体一句话说明为什么是他们。

注意区分：
- **买单者**（付钱的人）和 **使用者**（日常用的人）—— B2B 里这经常不是同一个人
- 垂直人群的特征越具体越好（不要只说"中小企业"，要说"没有专业投手的小电商卖家"）

### 3. 为什么有人付钱

分析用户的支付动机。回答三个问题：

1. **什么痛点？** 痛点越具体、越痛，用户越愿意付钱
2. **痛多久了？** 长期持续的痛比一次性痛更有商业价值
3. **痛到愿意掏多少钱？** 判断价格敏感度

参考维度：
- 金钱损失型（广告浪费、买错零件、被罚款）→ 最愿意付钱
- 时间损失型（重复劳动、低效流程）→ 中等
- 焦虑消除型（找工作、育儿、陌生环境）→ 短期高意愿
- 身份提升型（写得更好、变得更专业）→ 低但持续

**核心判断：** 用户不是在买功能，是在买**确定性**——确定广告不被浪费、确定简历能被选中、确定明天能钓到鱼。

### 4. 当前用户如何解决（产品不存在时）——事实调研

**这是最重要的新增步骤。** 不调研用户现状的分析是空中楼阁。在评估产品可行性时，必须回答：**没有这个产品，用户现在怎么解决？有多痛？**

#### 调研要求

不要凭空想象。必须用 WebSearch 查找**真实的事实依据**：

1. **用户现在的替代方案：** 列出每个方案以及对应的代价（时间/金钱/风险）
2. **替代方案的关键缺陷：** 每个方案具体差在哪
3. **痛苦程度的量化证据：** 寻找真实的调研数据、论坛帖子、用户评价、成本数据

#### 事实依据的获取来源

| 来源 | 找什么 | 示例 |
|------|--------|------|
| 行业调研报告 | 付费意愿、痛点排名、市场数据 | FreshBooks 2025: 35% 企业主说"整理收据"是#1 头疼 |
| Seller Central / Reddit 论坛 | 真实用户抱怨、损失案例 | Amazon 卖家论坛: "标签错误"致 $8,000 货物被吞 |
| Capterra / G2 产品评测 | 竞品的用户差评、切换原因 | Prisync: "45% 静默屏蔽率，返回旧数据但不报错" |
| 行业研究机构 | 品类规模的量化论文 | McKinsey: 高管 61% 认为决策时间被浪费 |
| IndieHackers / Dev.to | 个人创业者的自述实践 | "自建爬虫每 2-3 周崩一次，CAPTCHA 率 30-40%" |
| 学术论文 | 权威的经济效应数据 | Cambridge: 信贷员疲劳决策每月损失 $509K |
| 产品官网 / 定价页 | 竞品功能覆盖和价格 | 对比定价和功能边界 |

#### 输出格式

```
## 🔍 当前用户如何解决（无 XXX 时）

| 方案 | 代价 | 痛点 |
|------|------|------|
| [方案 1] | [时间/金钱成本] | [关键缺陷] |
| [方案 2] | ... | ... |

## 📊 痛苦程度事实依据

### [关键数据]

| 数据点 | 数值 | 来源 |
|--------|------|------|
| 核心痛点数据 | ... | 来源链接 |
| ... | ... | ... |

### 真实用户案例

> "直接引用的论坛帖子或用户投诉"——来源

### 替代方案缺陷分析

[现有方案为什么不够好的客观分析]
```

#### 检查清单

完成调研后，回答这几个问题：
- [ ] 用户的替代方案是否列全了？
- [ ] 每个替代方案的代价是否量化了？
- [ ] 有没有找到至少 2-3 个独立的来源交叉验证？
- [ ] 证据是否来自一手用户反馈（论坛/评测）而非推论？
- [ ] 是否诚实标注了"没有找到直接数据"的情况？

#### 错误做法

- ❌ "用户很痛苦"——必须带证据
- ❌ "这是常识"——常识经常是错的，找数据
- ❌ 只用一篇文章/一个来源——交叉验证
- ❌ 忽略竞品的存在——"没有竞品"通常等于"没有市场"

### 5. 护城河

这个产品的真正护城河是什么？提醒：技术通常不是护城河。

常见的真正护城河：
- **权限信任**：别人不敢轻易把 Gmail、Google Ads、银行账户交给新来的
- **领域知识**：不懂行业做不出来的产品（CrowdField、Marky）
- **嵌入程度**：嵌进用户现有工作流越深，替换成本越高（Checker Plus 嵌进 Gmail 习惯）
- **长期评价**：App Store 几万个好评、Chrome Web Store 长期积累
- **网络效应**：用户越多产品越好（marketplace、社区）
- **分发权重**：SEO 积累、ASO 排名、媒体背书
- **数据飞轮**：使用越多数据越准（FishingAI、Gear Snap）

如果发现产品没有真正的护城河，直说。

### 6. 商业模式

简要分析：
- **定价方式**：月订阅 / 按量 / 一次性 / 免费+增值 / 佣金 / affiliate
- **获客渠道**：SEO / ASO / 社媒 / paid ads / 分销 / 社区 / 内容营销
- **单位经济**：估算客单价、毛利率（如果能从信息推断）
- **增长瓶颈**：什么在限制它增长（分发、产能、地域、法规）

### 7. 风险与脆弱性

诚实判断这个产品最脆弱的地方。

常见脆弱点：
- **平台依赖**：寄生在大平台上，平台一变就死
- **竞争挤压**：大厂进场 / 免费竞品 / commodity 化
- **生命周期短**：用户只用一段时间（育儿、求职）
- **合规风险**：隐私法规、灰色地带、平台封号
- **准确性风险**：识别错了、建议错了，用户直接流失
- **运营太重**：需要人工交付、供应商管理、本地服务
- **泛化问题**："什么都做"但"什么都不深"，用户记不住你

### 8. 你能学什么

这是最重要的部分。不要只复述上面的分析，要给出**可复用的创业方向抽象**。

模式：
1.  指出这个产品属于哪一类（见下面的分类体系）
2.  抽象出它的**骨架**——不是克隆它，是提取它成功的本质
3.  给出 2-3 个**具体可做的衍生方向**

### 9. （可选）衍生创业方向

如果合适，用这个公式给出一个创业方向建议：

> **垂直行业 + 明确金钱损失 + AI 自动执行 + 月订阅**

格式：
```
**方向名称：** [一句话描述]
**适用人群：** [具体]
**为什么能赚钱：** [具体]
**切入点：** [第一步怎么做]
```

## 分类体系

分析完成后，把产品归入以下类别之一（可能跨类）：

### 第一类：结果型 AI agent
不是回答问题，而是完成工作。
代表：RankSpot、NotFair、Synta、Collabute
特征：帮用户少花时间、少亏钱、减少错误

### 第二类：垂直专业工具
切中具体行业/人群的深度工具。
代表：CrowdField、Marky、FishingAI、Gear Snap、Mommy
特征：用户具体，痛点具体，语言具体；越具体越好卖

### 第三类：浏览器扩展型工作流插件
寄生在大平台上的轻量工具。
代表：Checker Plus、BeLikeNative、Word Spinner
特征：开发轻、嵌入深、分发清晰；平台依赖重

### 第四类：App Store 订阅小工具
移动端专一功能工具。
代表：Framera、FishingAI、Mommy、BambooVPN、Gear Snap
特征：拼 ASO 和转化，不拼技术

### 第五类：小资产 marketplace / 数据平台
聚合、验证、交易产品。
代表：TrustMRR / Unicorne
特征：不生产产品，生产发现和交易产品的场景

## 信息获取策略

当用户给出产品名称时：

1. 先用 WebSearch 搜索产品的基本信息（官网、MRR、用户评价）
2. 如果产品信息不充分，诚实标注"基于有限信息分析"
3. 对于知名产品（如 Shopify、Canva），不需要搜索，直接分析
4. 如果用户只给了产品类别或想法（如"做一个给装修公司找客户的工具"），按框架分析这个方向的可行性

### 痛苦事实调研专用搜索策略（第 4 步用）

搜索痛点事实依据时，使用以下搜索策略：

**按目标搜索：**
- `[品类/问题] + "pain point" OR "frustration" + 2025` — 找英文论坛帖
- `[工具名] + "reviews" OR "alternatives" + site:capterra.com` — 找竞品差评
- `[问题] + "cost" OR "time" OR "hours per week"` — 找量化数据
- `"I wish" OR "why can't" + [场景]` — 找用户自发表达的需求
- `Reddit + [行业] + [痛点]` — 找真实用户自述

**交叉验证原则：**
- 至少找到 2-3 个独立来源
- 区分"数据"和"观点"——带数字的优先
- 标注每个数据的来源链接
- 找不到直接数据时，诚实标注"缺乏该方向的专项统计数据"

### 输出格式中的事实标注

在分析中引用事实时，使用 markdown 脚注或行内引用格式：

```
> 来源：[来源名称](URL)
```

对于关键结论的支撑，至少提供：
- **量化数据**：百分比、金额、时间消耗（如有可能）
- **用户原话**：论坛/评测中的直接引用
- **案例**：具体的事件或损失案例

## 分析态度

- **诚实 > 好评。** 如果产品没有护城河，说没有。如果方向不行，说不行。
- **具体 > 笼统。** "帮小企业减少广告浪费" 好过 "帮企业提升效率"
- **深刻 > 全面。** 宁愿抓住 3 个真洞察，不要写 10 条废话
- **可操作 > 可读。** "你能学什么" 部分必须给出具体方向
